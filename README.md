# BreathaTech 
## What It Is

BreathaTech is an AI-powered clinical decision support system for a handheld breath-based chemical triage device. A patient exhales into the device, electrochemical sensors measure breath biomarkers, and an ML model classifies the likely exposure agent and severity — then surfaces a treatment recommendation to first responders.

**Agents detected:** Carbon monoxide (CO), nerve agents (organophosphate class: sarin, soman, VX), and phosgene.

---

## How to Run

### Prerequisites

```bash
pip install -r requirements.txt
```

### 1. Train the models (first time only)

The trained model files are already checked in to `backend/models/`, so you only need this step if you want to retrain from scratch:

```bash
cd backend
python train_models.py
```

### 2. Start the backend API

The API must be started from inside the `backend/` directory because `api.py` imports `inference` as a local module:

```bash
cd backend
 python -m uvicorn api:app --reload --port 8000   
```

### 3. Start the frontend

In a separate terminal:

```bash
cd frontend
python3 -m http.server 3000
```

Then open `http://localhost:3000` in a browser.

### 4. Make a prediction

1. Click a preset in **Device transmission** (CO / Moderate, Nerve Agent, Phosgene Occult, Baseline) to load a real training-data row into the sensor cards — or click **Manual input** to type your own sensor values directly into the sensor panel.
2. Adjust patient vitals, select signs and symptoms, and add clinical notes as desired.
3. Click **Submit to ML model**.
4. Review the triage recommendation, agent probabilities, and treatment guidance.

### API endpoints

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/predict` | POST | Run inference on a patient reading |
| `/health` | GET | Service health check |
| `/model/info` | GET | Model metadata and version |

## How It Works

### Data pipeline

1. **Frontend input** ([frontend/](frontend/)) — JavaScript UI provides sensor readings (eCO, eNO, eCO₂, OP score) two ways: via preset demo cases loaded from `data/demo_cases.json` (real rows from training data), or via **Manual input** mode, which swaps the sensor cards for editable number fields. Each field shows its normal reference range and valid entry range (0–max). Values update the ML probability bars live. Sensor data is combined with clinician-entered data (vitals, symptoms, demographics, `time_since_exposure_min`) into a single `reading` object and POSTed to the backend.
2. **API** ([backend/api.py](backend/api.py)) — FastAPI receives the `PredictRequest`, validates it against the `SensorReading` schema, and calls inference.
3. **Inference** ([backend/inference.py](backend/inference.py)) — `BreathaTechInference` loads the trained models and runs `predict()`. Returns agent class, severity class, and clinical alert flags.
4. **Frontend output** — the JavaScript results panel renders the ML output, treatment hint, and any critical alerts (`nerve_alert`, `phosgene_occult`).

### Sensors and why each biomarker is detectable in breath

| Sensor | Label | Normal ref | Entry range | Clinical rationale |
|--------|-------|-----------|-------------|-------------------|
| eCO | Exhaled CO | < 6 ppm | 0–80 ppm | Tracks carboxyhemoglobin (HbCO) in real time; corrects the SpO₂ paradox — pulse oximetry reads falsely normal in CO poisoning. Smokers baseline 15–20 ppm. |
| eNO | Exhaled NO | < 25 ppb | 0–120 ppb | Rises 30–60 min into the phosgene latent phase before fatal pulmonary edema develops — the only window to intervene. Measured via direct oxidation, Pt/Nafion electrode. |
| eCO₂ | Exhaled CO₂ | 4.0–4.5% | 0–7% | Reflects ventilatory distress; elevated in OP and phosgene, correlates with severity. |
| OP Score | Organophosphate index | < 5 | 0–100 | Unitless composite score derived from an electrochemical aptamer-based (E-AB) biosensor on an Au electrode with methylene blue reporter. Near-zero for CO, phosgene, and baseline patients; scales with OP severity (mild ≈ 18, moderate ≈ 48, severe ≈ 82). Models a point-of-care cholinesterase-activity readout — critical because the pralidoxime reactivation window closes in ~2 min for soman, ~5h for sarin, ~40h for VX. |

**Why HCN, Cl₂, and biologics are excluded:** HCN washes out in 10–22 seconds (patient is incapacitated before a breath sample is possible). Cl₂ is consumed in the airway and not exhaled. Biologics (anthrax, tularemia, plague, TB) operate on a days-to-weeks detection timescale — wrong device, wrong problem.

### ML models

Two XGBoost classifiers live in [models/](backend/models/):

- **Agent classifier** (`agent_model.pkl`) — 4-class: CO / OP / PHOSGENE / NONE. AUC 0.9999.
- **Severity classifier** (`severity_model.pkl`) — 3-class: severity 1 / 2 / 3. AUC 0.9986.

Training data is synthetic, generated by Monte Carlo simulation ([model_v5/generate_data.py](model_v5/generate_data.py)) — 2,400 patients with data derived from clinical literature on each agent's physiological signature.

## Why the Model Works the Way It Does

### XGBoost, not a neural net

The feature space is tabular and relatively small (sensor readings + vitals + symptoms + demographics). Tree-based gradient boosting consistently outperforms neural nets on structured tabular data at this scale. XGBoost is also fully explainable per-prediction via SHAP values, which matters for clinical deployment — a first responder or medical director needs to understand why the model flagged an agent, not just that it did.

### `exposure_ppm` is intentionally absent

 A feature the device cannot observe cannot be an input to the model. Removing it forces the classifier to work only from what the sensor and clinician can actually provide.

### `time_since_exposure_min` is clinician-entered, not measured

The device cannot know when exposure occurred. This value is entered by the clinician and defaults to 30 minutes if unknown. It must be set to 0 for true baseline (no-exposure) cases. It is load-bearing for the OP class — the nerve alert flag (`nerve_alert`) uses it to flag cases where the pralidoxime window may already be closing.

### Two separate models, not one

Agent classification and severity classification are decoupled. A single multi-output model would couple the two tasks and make it harder to update severity thresholds independently as clinical guidance changes. The two-model approach also lets the severity model condition on features that are only meaningful once an agent is identified.

### Clinical alert flags are rule-based, not model outputs

`nerve_alert` and `phosgene_occult` are deterministic rules applied on top of model output — they are not predicted by the classifier. This is intentional: these flags represent hard clinical constraints where a miss is catastrophic, and they must be transparent and auditable. `nerve_alert` fires when the model predicts organophosphate exposure and `time_since_exposure_min` is within the pralidoxime window. `phosgene_occult` fires when the model predicts phosgene and severity is low — the patient looks stable but is not.

### Smoker baseline adjustment

Smokers have a resting eCO of 15–20 ppm, which overlaps with mild CO poisoning in non-smokers. The `is_smoker` flag is a model feature so the classifier learns this distribution shift from training data rather than requiring a hard-coded threshold correction.

### SpO₂ is in the feature set but flagged as unreliable for CO

SpO₂ is included as a feature because it is informative for severity in OP and phosgene cases. However, the clinical rules explicitly prohibit surfacing SpO₂ as a reassuring signal alongside elevated eCO — pulse oximetry cannot distinguish oxyhemoglobin from carboxyhemoglobin. This constraint is enforced in the UI layer, not the model.

## Treatment Reference

| | Severity 1 | Severity 2 | Severity 3 |
|---|---|---|---|
| CO | 100% O₂ NRB mask | ECG + hyperbaric O₂ if HbCO >25% | HBO + ICU |
| OP | Atropine 2mg + pralidoxime 1g IV | Atropine titrated + 2-PAM STAT + diazepam | Full airway management + ICU |
| PHOSGENE | Strict rest + supplemental O₂ + admit 24h | ICU monitoring | Mechanical ventilation |
